param_t¶

A structure param_t is defined in include/param.h, which contains general parameters which are referred to by other functions:

28struct param_t_ {
29  // restart / initialise
30  bool load_flow_field;
31  char *dirname_restart;
32  // fluid-temperature coupling
33  bool with_temperature;
34  bool with_thermal_forcing;
35  // domain sizes
36  int itot, jtot;
37  double lx, ly;
38  // grid sizes
39  double stretch;
40  double *xf, *xc;
41  double *dxf, *dxc;
42  double *yf, *yc;
43  double dy;
44  // non-dimensional params
45  double Ra, Pr;
46  // temporal integration
47  rkcoef_t rkcoefs[3];
48  double time, dt;
49  int step, implicitx, implicity;
50  double expimp_wtimes[2][2];
51  // when to stop, when to write log, etc.
52  double timemax, wtimemax;
53  schedule_t log, save, stat;
54};

The meanings of each member are as follows:

  1. Initialisation: load_flow_field

    Whether flow fields (velocity, temperature, etc.) are loaded (true) or not (false). If not, they are initialised; otherwise loaded from a specified directory param->dirname_restart.

  2. Temperature coupling

    • with_temperature

      Whether the temperature field is solved (true) or not (false) as well as the flow fields (velocity and pressure).

    • with_thermal_forcing

      Whether the buoyancy force is added (true) or not (false) to the momentum balance in \(x\) direction (see the governing equations). If not, the temperature is a passive scalar merely transported (advected and diffused) by the fluid motion.

  3. Domain size and coordinates

    • itot and jtot

      Numbers of cell centers (where pressure and temperature are defined) in \(x\) and \(y\) directions, respectively.

    • lx and ly

      Physical domain sizes (lengths) in \(x\) and \(y\) directions, respectively. lx is fixed to 1 since the governing equations assume \(l_x \equiv 1\) when being normalised (see the governing equations).

    • stretch

      The amount of distortion of the grid in \(x\) direction. Roughly, the smaller (minimum is 0) this number is, the more stretched the grid is and more points exist in the vicinity of the walls. See set_coordinate in src/param/init.c.

    • xf and xc

      Coordinates in \(x\) direction. xf and xc are used to define cell-face and cell-center locations, respectively. See set_coordinate in src/param/init.c.

    • dxf and dxc

      Distance between neighbouring param->xf and param->xc, respectively. See set_coordinate in src/param/init.c.

    • dy

      Grid size in \(y\) direction. Since only uniform grid arrangements are allowed for now, a scalar value is sufficient. yf and yc are, however, defined since they are useful for post-processings.

  4. Non-dimensional parameters

    • Ra

      Rayleigh number (see the governing equations)

    • Pr

      Prandtl number (see the governing equations)

  5. Parameters relevant to temporal integration

    • rkcoefs

      Coefficients of Runge-Kutta scheme \(\alpha,\beta,\gamma\) (see Temporal discretisation)

    • time

      Simulation time (not wall time)

    • dt

      Time step size

    • step

      Number of steps (how many iterations have been done up to now)

    • implicitx, implicity

      Whether the diffusive terms in each direction are treated implicitly (1) or not (0). See the temporal discretisation and src/param/decide_dt.c.

    • expimp_wtimes

      Variable to store wall times which are needed to complete the whole procedures with explicit or implicit treatments of the diffusive terms. See src/param/estimate_cost.c.

  6. Schedulers

    • timemax

      Maximum simulation time (in simulation time units). Simulation is terminated if param->time exceeds this parameter (see src/main.c).

    • wtimemax

      Maximum simulation time (in wall time). Simulation is terminated if param->wtime exceeds this parameter (see src/main.c).

    • log, save, stat

      Decide timings of logging, saving flow fields, and collecting statistics, respectively. Each member has three values:

      1. rate

        This process is repeated in each rate time units. This is needed since (generally) we do not want to dump flow fields (MB or GB in sizes) every time step.

      2. after

        This process is only executed after after time units. This is needed since (generally) we are only interested in flow statistics after the flow reaches the statistically-steady state.

      3. next

        This process will be executed at next. This is used as a timer to trigger the next event.

Although default values are assigned, these parameters can be specified as environmental variables and one can easily override them (see exec.sh). When being given, the parameters are loaded by load_config in src/param/init.c.
